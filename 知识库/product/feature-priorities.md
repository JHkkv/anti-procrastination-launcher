# 功能优先级矩阵：先动起来（Anti-Procrastination App）

> 分析日期：2026-07-08
> 基于竞品分析 + 用户画像推导

---

## 一、功能优先级总览

| 优先级 | 含义 | 数量 | 开发周期预估 |
|--------|------|------|-------------|
| **P0** | 必须有 —— 没有它产品不成立 | 6 个功能 | 4-6 周 |
| **P1** | 应该有 —— 显著提升体验和留存 | 8 个功能 | 3-4 周 |
| **P2** | 可以有 —— 差异化竞争点 | 6 个功能 | 2-3 周 |

## 二、P0 —— 必须有（MVP 核心）

> 没有这些功能，产品价值主张不成立。

### P0-1：AI 任务拆解

| 维度 | 详情 |
|------|------|
| **功能描述** | 用户输入一个任务（如"写论文"），AI 自动拆解为可执行的微步骤（如"打开文档→写标题→写第一段大纲"） |
| **用户价值** | 解决核心痛点——拖延者不知道从何开始。覆盖全部4个画像 |
| **开发复杂度** | 中 |
| **实现方案** | 调用 LLM API（Claude/GPT），Prompt 模板化；本地缓存常见任务拆解结果 |
| **技术栈** | 前端输入框 + 后端 API 调用 + 结果展示列表 |
| **预估工时** | 5-7 天 |
| **风险** | LLM 响应延迟（需优化 UX 如 streaming 输出）；成本控制 |
| **技术适配** | 主方案：Claude API / GPT API；国内备选方案：百度文心一言（ERNIE）、阿里通义千问（Qwen）、字节跳动豆包（Doubao）。部署建议：优先选用国内大模型以降低延迟（无需跨境网络）和满足数据合规要求（《个人信息保护法》）；Prompt 模板需针对中文语境优化；本地缓存常见任务拆解结果以降低 API 调用成本 |

### P0-2：只看下一步（Next-Step UI）

| 维度 | 详情 |
|------|------|
| **功能描述** | 屏幕上只显示当前要做的一步，完成后自动切换到下一步。隐藏完整任务列表 |
| **用户价值** | 降低视觉压力和决策负担——不让用户看到"还有20步"。ADHD 用户的刚需 |
| **开发复杂度** | 低 |
| **实现方案** | 状态机：当前步骤 → 完成 → 下一步。全屏卡片式 UI |
| **技术栈** | 前端组件（React/Vue/原生） |
| **预估工时** | 3-4 天 |
| **风险** | 需要平衡"只看一步"与"看全局进度"的需求（P1 中的进度条缓解） |

### P0-3：极简启动（2 分钟法则）

| 维度 | 详情 |
|------|------|
| **功能描述** | "先做2分钟试试"模式——用户只需承诺做2分钟，2分钟后可选择继续或停下 |
| **用户价值** | 利用行为启动理论——开始比坚持更容易。降低拖延者的启动焦虑 |
| **开发复杂度** | 低 |
| **实现方案** | 倒计时组件 + 完成/继续的二选一弹窗 |
| **技术栈** | 前端 Timer 组件 |
| **预估工时** | 2-3 天 |
| **风险** | 2分钟后用户停下的比率高——需设计"继续"的引导话术 |

### P0-4：零配置首页

| 维度 | 详情 |
|------|------|
| **功能描述** | 打开App即见今日任务概览 + 第一个待执行步骤，无需任何配置 |
| **用户价值** | 竞品最大的问题——Notion/Habitica 都需要花时间配置。零配置=零拖延入口 |
| **开发复杂度** | 低 |
| **实现方案** | 默认视图设计；引导式 onboarding（3步内完成） |
| **技术栈** | 前端 UI/UX |
| **预估工时** | 3-5 天（含设计迭代） |
| **风险** | 首次使用如果没有任务，需要有默认示例任务 |

### P0-5：完成庆祝与正向反馈

| 维度 | 详情 |
|------|------|
| **功能描述** | 每完成一步，给予轻量正向反馈（动画、鼓励语、进度+1）；拒绝惩罚机制 |
| **用户价值** | 拖延者最不需要的就是被指责。正向反馈激活多巴胺回路，形成正循环 |
| **开发复杂度** | 低 |
| **实现方案** | Lottie 动画 + 鼓励语库（随机选取）+ 微震动反馈 |
| **技术栈** | 前端动画组件 |
| **预估工时** | 2-3 天 |
| **风险** | 动画/音效需要调性把控——太幼稚会让职场用户不适 |

> **"轻量正反馈"vs"过度游戏化"的边界说明**（回应心理学库关于过度理由效应的警告）：
>
> 本产品采用的是**轻量正反馈**（Lightweight Positive Feedback），而非重度游戏化（Heavy Gamification）。两者的区别如下：
>
> | 维度 | 轻量正反馈（本产品采用） | 过度游戏化（本产品避免） |
> |------|-------------------------|------------------------|
> | 反馈类型 | 完成动画、温和鼓励语、进度条推进 | 积分商城、等级系统、虚拟宠物、装备收集 |
> | 动机影响 | 增强胜任感（信息性反馈），不替代内在动机 | 可能引发过度理由效应，将内在动机转化为外在追逐 |
> | 惩罚机制 | 无——中断后温和重启 | 掉血、扣分、连续天数断裂惩罚 |
> | 理论依据 | Deci & Ryan：信息性反馈促进内在动机 | Lepper et al. (1973)：预期奖励削弱内在兴趣 |
>
> **设计红线**：任何新增的反馈元素都应通过以下检验——"如果去掉这个元素，用户的核心行为（开始任务）是否会改变？"如果答案是"不会"，说明该元素是信息性反馈（安全）；如果答案是"会"，说明该元素已变成外部控制（需要警惕）。

### P0-6：快速任务输入

| 维度 | 详情 |
|------|------|
| **功能描述** | 一个输入框就能创建任务——不用分类、不用设标签、不用设截止时间 |
| **用户价值** | 拖延者连"创建任务"这件事都会拖延。输入越少越好 |
| **开发复杂度** | 低 |
| **实现方案** | 单输入框，Enter 即创建。可选字段全部隐藏 |
| **技术栈** | 前端表单 |
| **预估工时** | 1-2 天 |
| **风险** | 过于简单的输入可能导致任务管理混乱——用 AI 拆解弥补 |

### P0 汇总表

| 功能 | 用户价值 | 开发复杂度 | 工时 | 优先依据 |
|------|---------|-----------|------|---------|
| AI 任务拆解 | 10/10 | 中 | 5-7 天 | 核心价值主张，无竞品做到 |
| 只看下一步 | 9/10 | 低 | 3-4 天 | 降低认知负担，ADHD 刚需 |
| 2 分钟法则 | 8/10 | 低 | 2-3 天 | 行为心理学支撑，极低开发成本 |
| 零配置首页 | 8/10 | 低 | 3-5 天 | 降低首日流失率 |
| 完成正向反馈 | 7/10 | 低 | 2-3 天 | 留存关键，与惩罚型竞品形成差异 |
| 快速任务输入 | 7/10 | 低 | 1-2 天 | 降低创建任务的拖延 |

**P0 总工时：约 16-24 天（3-5 周）**

## 三、P1 —— 应该有（增强体验）

> MVP 后第一批迭代功能，显著提升留存和 NPS。

### P1 功能列表

| 功能 | 用户价值 | 开发复杂度 | 工时 | 目标画像 | 说明 |
|------|---------|-----------|------|---------|------|
| **每日晨间 check-in** | 8/10 | 低 | 2-3 天 | 全部 | 打开App问"今天最重要的一件事是？"，帮用户聚焦 |
| **进度可视化（进度条/步数）** | 8/10 | 低 | 2-3 天 | 全部 | 看得到自己在推进（对抗"什么都没做"的感觉） |
| **情绪 check-in** | 7/10 | 低 | 2 天 | 职场/大学生 | "你现在感觉怎么样？"→焦虑时给更小的步骤 |
| **任务优先级排序** | 8/10 | 中 | 3-4 天 | 自由职业/ADHD | AI 根据 deadline 和紧急度自动排序 |
| **每日/每周回顾** | 7/10 | 中 | 3-4 天 | 全部 | "你今天迈出了8步"的正向总结（非"你还有5项"） |
| **自定义提醒时间** | 6/10 | 低 | 2 天 | 全部 | 智能推送：在用户最可能拖延的时间点提醒 |
| **番茄钟集成** | 6/10 | 低 | 2 天 | 职场/大学生 | 专注计时 + 休息提醒，但默认关闭（不增加压力） |
| **历史任务复用** | 5/10 | 低 | 2 天 | 自由职业 | 重复性任务的步骤模板一键复用 |

### P1 汇总表

| 指标 | 数值 |
|------|------|
| 总功能数 | 8 |
| 总工时 | 约 18-24 天（3-5 周） |
| 平均用户价值 | 6.9/10 |
| 对留存影响 | 预估提升 30-50% 次周留存 |

## 四、P2 —— 可以有（差异化竞争）

> 这些功能让产品从"好用"变成"离不开"。

### P2 功能列表

| 功能 | 用户价值 | 开发复杂度 | 工时 | 目标画像 | 说明 |
|------|---------|-----------|------|---------|------|
| **轻社交问责** | 7/10 | 中 | 5-7 天 | 大学生 | "好友也在动"的隐性社交压力，不是组队打怪 |
| **时间感知助手** | 8/10 | 中 | 4-5 天 | ADHD | "你已经做了15分钟"的提醒 + 时间进度条 |
| **脑倒（Brain Dump）** | 7/10 | 中 | 3-4 天 | ADHD/自由职业 | 随时输入想法 → AI 自动归类到项目和任务 |
| **上下文快照/恢复** | 7/10 | 高 | 5-7 天 | ADHD | 被打断时保存当前步骤，回来一键恢复 |
| **工作工具同步** | 6/10 | 高 | 7-10 天 | 职场新人 | 飞书/钉钉/日历的单向同步导入 |
| **能量水平追踪** | 5/10 | 中 | 3-4 天 | ADHD/自由职业 | 不同时段匹配不同难度任务 |

### P2 汇总表

| 指标 | 数值 |
|------|------|
| 总功能数 | 6 |
| 总工时 | 约 27-37 天（5-7 周） |
| 平均用户价值 | 6.7/10 |

## 六、开发路线图

### Phase 1：MVP（第 1-5 周）

```
目标：验证核心假设——"AI 拆解 + 只看下一步"是否能帮拖延者动起来

功能：P0 全部（6 个功能）
交付：可发布的最小可用产品
验证指标：
  - 日活用户次日留存率 ≥ 40%
  - 任务拆解使用率 ≥ 60%
  - 用户反馈中"帮我开始了"占比 ≥ 50%
```

### Phase 2：增强版（第 6-10 周）

```
目标：提升留存和使用深度

功能：P1 全部（8 个功能）
交付：体验完善的产品
验证指标：
  - 次周留存率 ≥ 25%
  - 日均使用时长 ≥ 10 分钟
  - NPS ≥ 30
```

### Phase 3：差异化（第 11-17 周）

```
目标：建立竞争壁垒

功能：P2 按需求排序选做（建议先做社交问责 + 时间感知）
交付：有独特价值的产品
验证指标：
  - 社交功能参与率 ≥ 20%
  - 用户推荐率 ≥ 15%
  - 付费转化率 ≥ 5%
```

### Phase 4：规模化（第 18 周+）

```
目标：扩展用户群和商业化

功能：P3 按需求排序选做
验证指标：
  - 月活突破 10 万
  - 付费用户 ARPU ≥ 50 元/月
```

## 七、功能依赖关系图

```
P0-6 快速任务输入
    ↓
P0-1 AI 任务拆解
    ↓
P0-2 只看下一步 ←──── P0-3（2分钟法则，可独立）
    ↓
P0-5 完成正向反馈
    ↓
P0-4 零配置首页（整合以上所有）

P1 依赖关系：
P0-1 → P1 任务优先级排序（需要先有任务）

```

## 八、商业可行性评估

### 定价策略建议

| 层级 | 价格 | 包含功能 |
|------|------|---------|
| **免费版** | 0 元 | 每日 3 次 AI 拆解 + 基础正向反馈 |
| **Pro 版** | 18 元/月 或 128 元/年 | 无限 AI 拆解 + P1 全部功能 + 数据回顾 |
| **团队版** | 48 元/人/月 | Pro + 社交问责 + 团队看板 |

### 成本估算（月）

| 项目 | 金额 | 说明 |
|------|------|------|
| LLM API | 0.3-0.8 元/活跃用户 | 取决于任务拆解频率和模型选择 |
| 服务器 | 500-2000 元 | 初期云服务，按量付费 |
| 总月成本（1000 用户） | 约 2000-5000 元 | 含人力外成本 |

### 关键假设

1. 拖延者愿意为"帮我开始"付费（需验证）
2. AI 拆解质量足够好，用户信任推荐（需 A/B 测试）
3. 轻量正向反馈比重游戏化更有效（基于心理学研究）
